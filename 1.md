# fpmul_xkh.c 修改记录

## 修复1：删除 debug 输出（原第164行）

**问题**：subnormal 分支中遗留了调试用的 `printf("%d %d\n", acount, a);`，导致输出中混入大量无关数字。

**修改**：删除该 printf 语句。

## 修复2：重写 subnormal 分支的移位和 GRS 计算（原第147~188行）

**问题**：原代码的 underflow 移位量计算有误。

```c
uint16_t delta = -exp - 14;
shift = delta - 2;
```

这个公式没有正确考虑 Raw 的 MSB 位置与目标 subnormal 格式之间的对齐关系。当 `shift` 为负数或过大时，后续的 `Raw >> shift` 和 sticky 位循环（`for(int16_t i = 9 + shift; i>=0; i--)`）会产生错误结果甚至溢出。

此外，原代码的 Action 判断也有 bug：

```c
if(sign == 1){
    Inexact = 0;  // 在打印 GRS 之前就把 Inexact 清零了
}
```

这导致负数 subnormal 的 Action 输出始终为 Truncate，但 GRS 的值可能不为零，逻辑上是对的（负数 round-to-+inf 确实 truncate），但代码结构容易引起混淆。

**修改**：重写整个 subnormal 分支。核心思路：

1. 计算净左移量 `net_left = (count + 1) + 14 + exp`，其中 `count` 是 Raw 的前导零数量，`exp` 是归一化后的指数。`count + 1` 是 normal 路径的归一化左移量，`14 + exp` 是从归一化位置调整到 E=-14 的偏移。
2. 如果 `net_left >= 0`，左移 Raw；否则右移，并将被移出的位累积到 sticky bit。
3. 从移位后的结果中按原代码的窗口格式（fraction=bit21..12, G=bit11, R=bit10, S=bit9..0）提取 GRS。
4. Action 判断统一为 `(Inexact && sign == 0) ? "Up" : "Truncate"`。

## 修复3：正常路径的 S 位计算错误（原第202~206行）

**问题**：原代码直接用 `Raw & FP16_S_MASK` 计算 S 位：

```c
if (Raw & FP16_S_MASK){
    S = 1;
}
```

这检查的是 Raw 原始值的低 10 位，但 S 位应该是归一化左移后（`Raw << (count+1)`）的低 10 位。当 Raw 的低 10 位恰好为 0 但左移后低 10 位不为 0 时（或反过来），S 位会算错。

例如 `7bff × 7bff`：Raw = 0x3FF001，Raw 低 10 位 = 1（S=1 是对的），但这只是巧合。正确做法应该检查移位后的值。

**修改**：改为检查 `shifted_raw & 0x3FF`：

```c
uint32_t shifted_raw = Raw << (count + 1);
uint16_t S = (shifted_raw & 0x3FF) ? 1 : 0;
```

## 修复4：正常路径的 Inexact 计算和 overflow 处理（原第208~253行）

**问题**：

1. Inexact 用了 `(Raw << (count+1)) & (0xffff - FP16_PRE_FRAC_MASK)` 这个 mask 运算，而不是简单的 `G | R | S`。这个 mask 计算不等价于 GRS 的 OR。

2. Overflow 时 GRS 和 Action 被硬编码：

```c
printf(" G=0 R=0 S=0 Action=Truncate\n");
```

但实际上 overflow 时 GRS 应该输出真实值，Action 也应该根据 Inexact 和 sign 判断。例如 `7bff × 7bff` 的 S=1, sign=0，Action 应该是 Up。

3. GRS 的打印在 rounding 和 overflow 判断之后，导致 overflow 时走不到正常的 GRS 打印逻辑。

**修改**：

1. 将 GRS 打印移到 rounding 之前，确保所有路径都能正确输出 GRS。
2. Inexact 改为 `G | R | S`。
3. Action 统一为 `(Inexact && sign == 0) ? "Up" : "Truncate"`。
4. Overflow 判断移到 GRS 打印之后，只输出 Result 行。
5. 最终 Result 的溢出检查阈值从 `>= 0x7bff` 修正为 `>= 0x7c00`（0x7BFF 是合法的最大正 normal 数，不应该被判为溢出）。
